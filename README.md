# Super Agent —— 企业级 AI 智能体对话平台

> 不是调个 API 就完事的 Demo，而是从对话入口到证据驱动生成、从文档治理到全链路观测，每一个环节都经过深度设计的完整工程实现。覆盖 Agent 智能体、RAG 检索增强生成、MCP 工具协议、Skills 能力扩展、会话记忆管理、文档全生命周期治理六大核心能力。

> [!IMPORTANT]
> **Super Agent 不是 Spring AI / Spring AI Alibaba 的套壳项目。** 框架提供的是模型调用、向量存储这些基础能力，但三层执行器路由、前置编排器五步决策链、双通道检索融合、Parent-Child 块聚合、证据预算控制与无证据短路、组合式切块引擎、摘要压缩记忆、集群级租约互斥、全链路观测追踪——这些构成项目核心竞争力的能力，全部是在框架之上从零设计和手写实现的。光靠框架远远撑不起一个企业级 AI 系统，真正的深度在框架之外。

# 一条消息背后发生了什么

用户在输入框里敲了一句话，点了发送。看起来很简单，但在 Super Agent 内部，这条消息要经过一条远比你想象中复杂的链路，才能变成一个靠谱的回答。

先看整体流程，后面再逐块拆解每个环节的设计细节：

<img src="https://multimedia-javaup.cn/super-agent/flow-chart/%E5%9B%BE%E4%B8%80%EF%BC%9A%E6%A0%B8%E5%BF%83%E5%AF%B9%E8%AF%9D%E6%80%BB%E8%A7%88.svg" align="left" width="50%" />

**第一步：流式订阅与集群锁定。** 请求到达后不会立即执行，而是等 SSE 连接建立后才启动处理链路（`Flux.defer`）。同时通过 Redis 分布式租约锁定当前会话，防止集群环境下同一条消息被多个实例重复处理。

**第二步：前置编排决策。** `ChatPreparationOrchestrator` 接管消息，依次完成路由判定、问题改写、子问题拆分、知识域收缩、歧义检测五个步骤，产出一份完整的 `ConversationExecutionPlan`——包含执行模式、改写后的问题、子问题列表、检索范围等所有决策结果。

**第三步：执行器分流。** 根据执行计划选择最合适的引擎：信息不全走歧义追问、知识类问题走 RAG 证据驱动生成、开放式问题走 ReAct Agent 自由探索。不是让模型自己决定所有事情，而是用确定性的编排逻辑做好决策后再交给执行器。

**第四步：流式输出与收尾。** 答案通过 SSE 实时推送，正文分片传输，结束时补发引用来源和推荐追问问题。无论成功还是失败，统一触发清理流程：释放集群锁、保存 Checkpoint、归档对话记录。

<!-- 📸 截图：对话的详细轨迹观测.png —— 一次完整对话的全链路追踪面板，展示思考过程、检索通道、证据来源等每个决策节点的详细记录 -->

这套链路的核心思路是：**把不确定性留给模型，把确定性交给工程。** 模型擅长理解语义和生成文本，但不擅长做路由决策和资源调度。所以 Super Agent 在模型介入之前，就已经把该做的决策全做完了。

# 三层执行器：不是所有问题都该交给 Agent

面试中聊 Agent，面试官不想听你说"我用了 ReactAgent"，他想听的是你对执行策略的思考。Super Agent 的核心设计理念是：**不同类型的问题，用不同的引擎来回答。**

## 为什么要分层？

知识问答追求"稳"和"可解释"——用户问"退款规则是什么"，你需要的是从文档中精准检索证据，然后让模型基于证据生成回答。这种场景让 Agent 自己探索反而容易不可控，可能跑去联网搜索一堆无关内容。

而"今天北京天气怎么样"这种问题，知识库里根本没有答案，必须让 Agent 调用搜索工具去获取实时信息。

所以 Super Agent 设计了三层执行器，按场景精准分流：

| 执行器 | 触发条件 | 处理方式 |
| :--- | :--- | :--- |
| 歧义追问执行器 | 用户问题信息量不足，无法确定意图 | 生成澄清问题，引导用户补充信息 |
| RAG 知识问答执行器 | 问题可以在知识库中找到答案 | 证据驱动生成，引用来源可追溯 |
| ReAct Agent 执行器 | 需要联网搜索、多步推理的开放式问题 | 自主决策 + 工具调用 + 多轮推理循环 |

判断顺序是：**歧义澄清 > 知识问答 > 开放式 Agent**。优先用最稳定的方式回答，只有确实需要自由探索时才启动 Agent。

## ReAct Agent 的安全边界

Agent 的自由度越高，失控的风险就越大。Super Agent 在 Agent 执行层做了多重防护：

- **模型调用次数限制**：`ModelCallLimitHook` 限制单次运行最多调用模型 8 次，单个会话线程累计上限 40 次
- **工具调用次数限制**：`ToolCallLimitHook` 限制搜索工具最多调 6 次，会话累计上限 30 次
- **工具重试与兜底**：`ToolRetryInterceptor` 做指数退避重试（最多 2 次，初始 200ms，最大 1200ms，带随机抖动），`ToolErrorInterceptor` 做最终兜底
- **并行工具执行**：最多 4 个工具并行执行，提升响应速度
- **Checkpoint 持久化**：对话状态存入 MySQL，应用重启后能继续之前的对话

<img src="https://multimedia-javaup.cn/super-agent/flow-chart/%E5%9B%BE%E4%B8%89%EF%BC%9AReAct%20Agent%20%E6%89%A7%E8%A1%8C.svg" align="left" width="50%" />

这些设计在代码里都能找到对应实现，不是概念上的东西。

# 前置编排器：模型回答之前的隐形决策链

系统不会把用户的原始问题直接扔给模型，而是先经过一轮完整的编排。这个编排器叫 `ChatPreparationOrchestrator`，它在每次对话中做的事情远比你想象的多。

<img src="https://multimedia-javaup.cn/super-agent/flow-chart/%E5%9B%BE%E5%9B%9B%EF%BC%9A%E5%89%8D%E7%BD%AE%E7%BC%96%E6%8E%92%E5%99%A8%E5%86%B3%E7%AD%96%E9%93%BE%E8%B7%AF.svg" align="left" width="50%" />

**路由判定** —— 用户发进来的消息，首先要判断它属于哪种类型。是知识库能回答的？还是需要联网搜索的？还是信息不全需要先追问？这个判定不是靠关键词匹配，而是通过 LLM 分析上下文后给出路由方向。

**问题改写** —— 用户的问题往往不适合直接检索。"那它怎么配置？"——"它"指的是什么？得结合前几轮对话才知道。`ChatQueryRewriteService` 把这些省略的信息补回来，让检索能找到东西。

**子问题拆分** —— "退款规则是什么？审批流程怎么走？"这种复合问题，如果直接拿去检索，两个意图互相干扰。系统会拆成独立子问题，每个单独走检索链路，最后合并结果。单轮最多拆 4 个子问题，避免过度切碎。

**意图解析与知识域收缩** —— 拆完子问题后，`ConversationIntentResolutionService` 分析每个子问题属于哪个知识域，把检索范围从"全库"缩小到"相关领域"。这一步直接影响检索的精准度和速度。

**歧义检测** —— 如果用户的问题信息量不够，比如只说"查一下那个"，系统不会硬着头皮去检索，而是先生成澄清问题让用户补充信息。这比返回一个不相关的答案体验好得多。

所有步骤完成后，编排器产出一份 `ConversationExecutionPlan`，包含执行模式、改写后的问题、子问题列表、知识域范围等完整决策上下文，交给对应的执行器去执行。

# 检索不是查个向量库那么简单

很多人以为 RAG 的检索就是"查个向量库 + 让模型回答"，但实际上中间的环节比想象中多得多。

<img src="https://multimedia-javaup.cn/super-agent/flow-chart/%E5%9B%BE%E4%BA%8C%EF%BC%9ARAG%20%E6%A3%80%E7%B4%A2%E9%93%BE%E8%B7%AF.svg" align="left" width="50%" />

## 双通道并行检索

用户问一个订单号、一个配置项名，向量检索很可能找不到——语义相似度对精确匹配天然弱势。所以 Super Agent 用了双通道并行：**向量检索走 PGVector，关键词检索走 Elasticsearch**，两路同时出发，互不阻塞。

两路结果的分数量纲完全不同，不能直接比大小。系统用 **RRF（Reciprocal Rank Fusion）** 按排名倒数法融合，向量通道设了最低相似度阈值 0.45，关键词通道用相对阈值 0.35，低于阈值的弱命中直接过滤。融合后还可以接**外部 Rerank 精排**（支持 SiliconFlow 兼容协议），在较干净的候选集上继续优化排序。

## Parent-Child 块聚合

这是整个检索链路中最精巧的设计之一。检索粒度用 Child 小块保证命中率——小块语义集中，更容易被向量检索命中。但回答阶段如果只用小块，上下文往往不完整。所以系统在命中 Child 块后，自动聚合提升到 Parent 大块，保证回答时有足够的上下文信息。

**检索用小块保精度，回答用大块保完整性。** 这个设计在业界也属于比较前沿的实践。

<!-- 📸 截图：查看父子块详细内容.png —— Parent-Child 块的层级关系展示，可以看到父块包含多个子块，以及各自的内容边界 -->

## 证据预算控制与无证据短路

模型的上下文窗口是有限的，不能把检索到的所有内容一股脑塞进去。Super Agent 做了严格的证据预算控制：

- 单个子问题最大 2200 字符
- 全部证据总预算 5200 字符
- 单个父块最大 2200 字符

如果最终没有任何有效证据，**直接短路返回**，告诉用户"当前文档中没有检索到足够证据"。不让模型凭空编造——这是从架构层面杜绝幻觉最直接的手段。

## 最终回答组装

按子问题边界分别组织证据，注入到 Prompt 里，模型按编号逐一回答。要求模型在引用证据时标注来源编号 `[1][2]`，答案通过 SSE 实时流式推送，结束时补发引用来源和推荐追问问题。

# 文档从上传到可检索的完整旅程

很多项目的文档处理就是"切成固定长度 → 向量化 → 完事"。但实际上不同文档差异很大，一刀切的效果很难好。Super Agent 的文档处理是一条完整的异步流水线，每一步都有独立的状态追踪和任务日志。

<img src="https://multimedia-javaup.cn/super-agent/flow-chart/%E5%9B%BE%E4%BA%94%EF%BC%9A%E6%96%87%E6%A1%A3%E5%A4%84%E7%90%86%E6%B5%81%E6%B0%B4%E7%BA%BF.svg" align="left" width="50%" />

## 第一步：上传与多格式解析

文件上传到 MinIO 对象存储后，通过 Kafka 异步触发解析任务。Apache Tika 3.2.3 负责处理 PDF、Word、PPT 等多种格式，把五花八门的文档统一转成干净的文本。PDF 里的表格、扫描件、双栏排版——每一个都是坑，Tika 帮你踩过了。

<!-- 📸 截图：文档列表.png —— 文档管理列表页面，展示已上传文档的名称、格式、解析状态、索引状态等信息 -->

## 第二步：智能策略推荐

解析完成后，系统不是让用户盲选切块算法，而是根据文档类型和内容特征**自动推荐最优的切块策略组合**。用户可以查看推荐结果，也可以手动调整——比如文档质量不太好，可以额外开启 LLM 智能切块。

<!-- 📸 截图：配置策略.png —— 切块策略配置界面，展示系统推荐的策略组合和用户可调整的参数选项 -->

## 第三步：组合式切块引擎

四种切块策略不是四选一，而是各司其职的组合流水线：

| 策略 | 角色 | 什么时候用 |
| :--- | :--- | :--- |
| 基于文档结构切块 | 主干 | 按标题/章节/段落切成语义完整的块 |
| 递归分块 | 兜底 | 结构块太大时继续往下裁剪，控制块大小 |
| 语义分块 | 优化 | 在结构切块基础上做边界精修 |
| LLM 智能切块 | 增强 | 处理低质量文档和复杂文档，默认关闭 |

一句话：**结构负责保留文档天然边界，递归负责控制块大小，语义负责优化块边界，大模型负责处理疑难场景。**

<!-- 📸 截图：验证分块结果.png —— 切块结果验证页面，展示每个块的内容、大小、所属层级，方便用户确认切块质量 -->

## 第四步：向量化与双引擎索引

确认策略后，系统通过 Kafka 异步执行向量化和索引构建。向量写入 PGVector，关键词写入 Elasticsearch，构建双通道检索的基础。每一步都有独立的任务日志，出了问题能精确定位到哪一步失败。

# 会话记忆：Token 成本和上下文完整性的博弈

20 轮对话全塞给模型？Token 成本扛不住。只带最近几轮？可能丢掉关键上下文。这是生产环境下绕不开的问题。

<img src="https://multimedia-javaup.cn/super-agent/flow-chart/%E5%9B%BE%E5%85%AD%EF%BC%9A%E4%BC%9A%E8%AF%9D%E8%AE%B0%E5%BF%86%E6%91%98%E8%A6%81%E5%8E%8B%E7%BC%A9.svg" align="left" width="50%" />

Super Agent 设计了三种策略，覆盖从简单到复杂的全部场景：

**无记忆** —— 每轮独立，不携带历史。适合一次性查询，Token 成本最低。

**滑动窗口** —— 保留最近 N 轮完整对话。适合短期连续追问，实现简单但长对话会丢失早期上下文。

**摘要压缩** —— 这是生产环境最推荐的方案。最近 4 轮原文始终保留不做压缩，更早的历史由 `PersistentConversationMemoryService` 增量摘要，单次最多推进 6 轮，避免一次处理超长历史。最近原文窗口最大 2200 字符，长期摘要最大 1400 字符。所有记忆数据持久化到 MySQL，应用重启不丢失。

<!-- 📸 截图：对话记录.png —— 多轮对话的历史记录界面，展示会话列表和对话内容的完整呈现 -->

`ai-example-spring-ai-memory` 模块里还有一个独立的三种策略对比演示，可以一次性看到同一组问题在三种策略下的回答差异。

# MCP 工具协议与 Skills 能力扩展

Agent 的价值不只是能聊天，更在于能**调用外部工具完成实际任务**。Super Agent 在工具集成这块做了两层设计，让 Agent 的能力边界可以持续扩展。

## MCP 工具协议

MCP（Model Context Protocol）是 Anthropic 提出的开放协议标准，定义了模型与外部工具之间的通信规范。Super Agent 基于 MCP 协议实现了完整的工具调用链路：

- **动态工具发现**：Agent 启动时自动扫描并注册可用的 MCP 工具，不需要硬编码工具列表
- **标准化通信**：支持 Stdio 和 SSE 两种传输模式，兼容主流 MCP Server 生态
- **参数自动提取**：模型根据用户意图自动识别需要调用的工具并提取参数
- **多工具编排**：单次对话中可以串联调用多个 MCP 工具，前一个工具的输出可以作为后一个工具的输入
- **安全沙箱**：工具调用在受控环境中执行，有超时限制和异常隔离

相比传统的 Function Call 硬编码方式，MCP 协议的优势在于**标准化和可扩展性**——新增一个工具不需要改任何业务代码，只需要部署一个符合 MCP 协议的 Server 即可。

## Skills 能力扩展

Skills 是比 MCP 更上层的能力抽象，解决的是"Agent 怎么获得特定领域的专业能力"这个问题：

- **声明式定义**：每个 Skill 通过 `SKILL.md` 配置文件描述能力边界、触发条件、执行逻辑
- **目录化管理**：Skills 按领域组织成目录结构，支持嵌套分类
- **自动加载机制**：系统启动时自动扫描 Skills 目录，新增 Skill 只需放入对应目录，零配置生效
- **引用脚本与参考资料**：Skill 可以关联外部脚本和参考文档，执行时自动加载上下文
- **能力组合**：多个 Skills 可以组合使用，Agent 根据任务需求自动选择最合适的 Skill 组合

Skills 体系让 Agent 的能力边界不再是固定的。今天加一个"数据分析"Skill，明天加一个"代码审查"Skill，Agent 的能力就跟着增长，核心代码一行不用改。

# 工程质量不是靠嘴说的

说一个项目是企业级，得看实际的工程质量。从几个维度来评估 Super Agent：

## 分层架构

项目采用多模块 Maven 架构，按职责清晰分层：

- **super-agent-business**：AI 业务核心，包含对话、检索、文档处理、Agent 执行等全部业务逻辑
- **super-agent-common**：通用 Web 能力层，统一异常处理、响应格式、公共字段自动填充
- **super-agent-redisson-framework**：分布式基础设施，Redis 分布式锁、租约管理、并发控制
- **super-agent-id-generator-framework**：分布式 ID 生成服务

AI 业务逻辑、通用 Web 能力、基础设施组件互不耦合。换检索引擎不用改业务代码，换分布式锁实现不用动对话逻辑。

## 集群安全与并发控制

这是很多开源项目完全忽略的部分，但在生产环境中至关重要：

- **Redis 租约互斥**：`RedisLeaseManager` 实现集群级别的会话锁定，防止同一条消息被多个实例重复处理
- **JVM 级任务注册**：`ChatRuntimeRegistry` 维护进程内的任务注册表，防止同进程重入
- **租约续期**：执行过程中自动续期，防止长对话超时导致锁释放后被其他实例抢占
- **优雅降级**：无论成功还是失败，统一触发清理流程，不会留下孤儿锁

## 设计模式实战

项目中落地了多种经典设计模式，每个都解决了实际的扩展性或解耦问题：

| 设计模式 | 应用场景 | 解决的问题 |
| :--- | :--- | :--- |
| 策略模式 | 三种会话记忆策略、四种切块策略、多检索通道 | 不同策略可插拔替换，新增策略不改已有代码 |
| 工厂模式 | 检索通道创建、切块器创建、执行器选择 | 复杂对象的创建逻辑集中管理 |
| 模板方法 | 文档处理流水线各节点 | 统一执行流程，子类只关注核心逻辑 |
| 责任链模式 | 前置编排器五步决策链、工具拦截器链 | 多个处理步骤按顺序串联，灵活组合 |
| 观察者模式 | SSE 流式输出回调 | 流式事件的异步通知与推送 |
| 注册表模式 | 执行器注册、工具注册、策略注册 | 组件自动发现与注册，新增组件零配置 |
| AOP | 重复执行限制、全局异常拦截、链路追踪 | 横切关注点与业务代码解耦 |

## 可扩展性

核心模块都预留了扩展点，不需要改框架代码，加个实现类或配置文件就完事了：

- **新增检索通道**：实现检索通道接口，注册为 Spring Bean，自动参与双通道检索
- **新增切块策略**：实现切块策略接口，可加入组合流水线的任意位置
- **新增记忆策略**：实现记忆策略接口，系统自动识别并可配置使用
- **新增工具调用**：给 ReactAgent 注册新的 Tool，自动参与工具选择
- **新增 MCP Server**：部署符合 MCP 协议的 Server，Agent 自动发现并注册
- **新增 Skill**：在 Skills 目录下放入 SKILL.md 配置文件，零配置即刻生效

## 全链路可观测

基于 AOP 的全链路追踪，每个环节的耗时、输入输出、决策结果都有记录。思考过程、检索通道使用情况、证据来源、工具调用记录——全部可视化呈现在管理后台的观测面板中。出了问题不用猜，直接看 Trace 就知道哪一步出了问题。

# 为什么选择开源

之所以选择开源，原因很简单：**对项目质量足够自信。** 架构设计、代码实现、工程规范，每一行都经得起审视。不藏着掖着，好不好你 clone 下来自己看——目录结构、提交记录、注释规范，全是明牌。

市面上不少项目只敢放几张截图、讲几个概念，真正敢把代码全部摊开的并不多。Super Agent 敢这么做，是因为前面讲的那些能力——三层执行器、前置编排器、双通道检索、Parent-Child 块聚合、证据预算控制、集群安全——不是 PPT 里的架构图，是你能跑起来、能断点调试、能逐行阅读的真实代码。

开源对你来说意味着什么：

- **源码即文档**：想了解某个模块怎么实现的，直接翻代码，比任何教程都准确
- **本地可调试**：断点打到任意一行，跟着一次请求走完整个链路，比看架构图理解得深十倍
- **可参与贡献**：发现 Bug 提 Issue，有优化思路提 PR。参与一个企业级 AI 开源项目，本身就是简历上的亮点
- **持续迭代更新**：项目会持续演进，Star 和 Watch 之后能第一时间获取新特性

**如果觉得项目还不错，点个 Star 支持一下，这是对开源作者最好的认可！**
